# 테이블 설계 근거

카카오톡 기준 ERD의 각 테이블을 **왜 이렇게 나눴는지** 정리한 문서.
설계에 공통으로 적용한 원칙은 다음과 같다.

- **정규화**: 중복·이행 종속을 없애는 방향으로 테이블을 분해 (예: 이모티콘 패키지 분리)
- **상속 조인 전략**: 종류마다 컬럼이 다른 개체(메시지·채팅방)는 부모/자식 테이블로 분리
- **N:N은 조인 테이블로**: 회원↔채팅방, 회원↔이모티콘 패키지 같은 다대다 관계는 중간 테이블로 해소
- **자기참조**: 답장처럼 같은 개체를 가리키는 관계는 자기 자신을 FK로 참조
- **소프트 삭제**: 탈퇴해도 대화 맥락이 남아야 하므로 물리 삭제 대신 상태값으로 관리

---

## 1. 회원 · 친구

### users — 회원
- **역할**: 인증 정보와 프로필을 담는, 모든 관계의 중심 개체.
- **왜 이렇게**:
  - `email`을 UNIQUE로 둬서 로그인 식별자 중복을 막는다.
  - `status`(ACTIVE/INACTIVE/DELETED)로 **소프트 삭제**를 한다. 회원을 물리적으로 지우면 그 사람이 보낸 메시지·참여 기록의 참조가 깨지기 때문에, 탈퇴는 상태만 바꾼다.

### friends — 친구관계
- **역할**: 회원과 회원 사이의 친구 관계(요청·수락).
- **왜 이렇게**:
  - 회원↔회원은 다대다라 `users` 안에 넣을 수 없다. 관계를 별도 테이블의 한 행으로 만든다. (1NF)
  - `(user_id, friend_id)`에 유니크 제약을 걸어 **같은 상대에게 중복 친구요청**이 쌓이지 않게 한다.
  - `status`(PENDING/ACCEPTED)로 "요청 → 수락"의 상태를 관리한다.

## 2. 채팅방

### chat_rooms — 채팅방 (부모)
- **역할**: 여러 메시지를 담는 "공간". 채팅 하나하나가 아니라 **방**을 개체로 본다.
- **왜 이렇게**:
  - 채팅은 "정해진 방 안에서 메시지를 주고받는" 구조라, 방을 기준 개체로 두고 메시지가 방에 소속되게 한다.
  - `type`(SINGLE/GROUP/OPEN)으로 방 종류를 구분한다. 세 종류가 공통으로 갖는 속성만 여기 둔다. → **상속의 부모**.

### open_chat_rooms — 오픈채팅 전용 정보 (자식)
- **역할**: 오픈채팅방에만 필요한 추가 속성(방장, 정원, 소개, 입장코드).
- **왜 이렇게**:
  - 방장·정원 같은 컬럼은 1:1·그룹 방엔 필요 없다. 이걸 `chat_rooms`에 다 넣으면 일반 방에선 전부 빈 값이 된다.
  - 그래서 **오픈채팅일 때만 생기는 정보를 자식 테이블로 분리**하고, `chat_room_id`를 PK이자 FK로 부모와 잇는다. (조인 상속 전략)

### chat_participants — 채팅방 참여자
- **역할**: 어떤 회원이 어떤 방에 들어가 있는지.
- **왜 이렇게**:
  - 회원↔채팅방은 다대다(한 명이 여러 방, 한 방에 여러 명)라 **조인 테이블**로 해소한다.
  - `(room_id, user_id)` 유니크로 **같은 방에 중복 입장**을 막는다.
  - `last_read_message_id`로 "이 사람이 어디까지 읽었나"를 저장한다. 메시지마다 읽음 레코드를 만들면 폭발하므로, 참여자별로 **마지막 읽은 지점 한 값**만 두는 범위 방식을 쓴다.

## 3. 메시지

### chat_messages — 메시지 (부모)
- **역할**: 실제 대화 내용. 방 안에서 오간 메시지 본체.
- **왜 이렇게**:
  - `type`(TEXT/EMOJI/MEDIA)으로 메시지 종류를 구분한다. → **상속의 부모**.
  - `parent_message_id`가 같은 `chat_messages`의 id를 가리키게 해서 **답장**을 구현한다. 답장 대상도 결국 메시지이므로 자기 자신을 참조하는 게 자연스럽다. (없으면 NULL = 일반 메시지)
  - `sender_id`는 회원이 탈퇴해도 지우지 않는다. 대화 맥락을 유지하고, 발신자는 "알 수 없음"으로 표시한다.

### text_messages / emoji_messages / media_messages — 종류별 메시지 (자식)
- **역할**: 메시지 종류별로 필요한 값만 담는 자식 테이블.
  - `text_messages`: 본문(`content`)
  - `emoji_messages`: 어떤 이모티콘인지(`emoticon_id`)
  - `media_messages`: 파일 주소·종류·크기·썸네일
- **왜 이렇게**:
  - 텍스트엔 본문만, 이모티콘엔 이모티콘 번호만, 미디어엔 파일 정보만 필요하다. 서로 겹치지 않는다.
  - 이걸 `chat_messages` 한 테이블에 다 넣으면 종류마다 다른 컬럼이 전부 빈 값이 되고 무결성이 약해진다.
  - 그래서 공통 정보는 부모에, **종류별 정보는 자식에** 두는 조인 상속 전략을 쓴다. `message_id`를 PK이자 FK로 부모와 잇는다.
- **미디어는 한 테이블로**: 사진·카메라·파일·음성메시지는 모두 "파일을 담는다"는 점에서 같다. 종류마다 테이블을 만들지 않고 `media_messages.file_type`(IMAGE/VIDEO/FILE/AUDIO)으로만 구분해 하나로 처리한다.
- **두 종류의 이모지 구분**: 휴대폰 기본 유니코드 이모지(😀)는 별도 개체가 아니라 그냥 문자이므로 `text_messages.content`에 텍스트로 저장한다. (DB 문자셋 `utf8mb4` 필요) 반면 구매하는 카톡 캐릭터 이모티콘은 `emoji_messages`가 `emoticons`를 참조하는 방식으로, 서로 저장 경로가 다르다.

---

## 4. 이모티콘

### emoticon_packs — 이모티콘 패키지
- **역할**: 이모티콘 묶음(패키지)의 마스터 정보(이름, 퍼블리셔).
- **왜 이렇게**:
  - 개별 이모티콘마다 패키지명·퍼블리셔를 반복 저장하면, 하나를 고칠 때 여러 행을 다 고쳐야 하는 이행 종속이 생긴다.
  - 패키지 정보를 별도 테이블로 빼서 **3NF**를 만족시킨다.

### emoticons — 개별 이모티콘
- **역할**: 패키지에 속한 낱개 이모티콘.
- **왜 이렇게**:
  - `pack_id`로 소속 패키지를 참조만 한다. 패키지 정보 자체는 여기 중복 저장하지 않는다.

### user_emoticons — 이모티콘 소유 내역
- **역할**: 어떤 회원이 어떤 패키지를 구매(소유)했는지.
- **왜 이렇게**:
  - 회원↔패키지도 다대다(한 명이 여러 팩, 한 팩을 여러 명이)라 **조인 테이블**로 해소한다.
  - 이모티콘 전송 시 "이 사람이 이 팩을 실제로 갖고 있는지" 소유권을 검증하는 근거가 된다.
